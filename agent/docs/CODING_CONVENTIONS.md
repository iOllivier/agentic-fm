# FileMaker Coding Conventions — Hero Solution

Based on analysis of 437 scripts in the Hero database (music rights/clearance tracking).
Customized from [filemakerstandards.org/code](https://filemakerstandards.org/code) to match Greg's actual coding style.

> **CRITICAL — Hard Tabs for Indentation**
>
> All indentation inside calculations and expressions MUST use hard tab characters (ASCII 0x09), never spaces. This applies to `Let()` variables, `Case()` branches, `List()` arguments, and any nested function content that appears inside CDATA blocks in fmxmlsnippet output. AI-generated code must use literal `\t` tab characters for calculation indentation. In the FileMaker calculation dialog, use Option-Tab (Mac) or Ctrl-Tab (Win) to insert a tab.
>
> This rule applies to **calculation expressions only** — it does not govern HR script step indentation (which uses spaces for visual nesting in the editor).

These conventions apply to all FileMaker code generated by AI in this project — both scripts and calculation code. When any convention is ambiguous, prefer clarity and readability over brevity.

## Naming Conventions

### Variables

Greg's actual patterns (observed from 437 scripts):

| Scope | Prefix | Casing | Examples from Hero |
| --- | --- | --- | --- |
| Script-local | `$` | lowercase or camelCase | `$param`, `$counter`, `$loop_counter`, `$ids`, `$pub_uid`, `$lastError`, `$owner` |
| Global | `$$` | camelCase or snake_case | `$$user_name`, `$$user_id`, `$$full_name`, `$$email_address`, `$$fmVersion`, `$$SelectedIDs`, `$$navigation_allow_list` |
| Calculation-scoped (Let) | `_` (underscore) | lowercase short | `_p`, `_p2`, `_l`, `_l2` |

**Key differences from default conventions:**
- Greg uses `_` prefix for Let variables (NOT `~` tilde) — e.g., `_p`, `_p2`, `_l`
- Global variables use **snake_case** (e.g., `$$user_name`, `$$navigation_allow_list`) — NOT ALL_CAPS with dots
- Script-local variables mix camelCase and snake_case — both are acceptable
- Short, pragmatic variable names preferred over verbose descriptive ones

### Tables

- Underscore-separated words: `Writer_Tracker`, `Artist_Tracker`, `Catalog_Writer_Pub_linker`
- Single word when sufficient: `Tracks`, `Artists`, `Publishers`, `Notes`, `Addresses`
- NOT TitleCase single word — multi-word uses underscores

### Fields

Greg's actual field naming patterns:

| Type | Convention | Examples from Hero |
| --- | --- | --- |
| Regular data | snake_case or camelCase | `track_uid`, `track_ISRC`, `Artist_full_name`, `priority_type`, `Publisher_uid` |
| Primary key | `uuid` or descriptive `_uid` | `uuid`, `Track_UID`, `Publisher_uid`, `Employee_uid` |
| Foreign key | `reference_uid` or descriptive | `track_uid`, `artist_uid`, `user_uid` |
| Auto-enter audit | `z` prefix | `zCreatorAccount`, `zModifierAccount`, `zTimestampCreated`, `zTimestampModified` |
| Utility/system | `zzz_` prefix | `zzz_found_count`, `zzz_record_number`, `zzz_quick_find`, `zzz_record_id`, `zzz_i_can_view` |
| Temp/global UI | `zzz_temp_` prefix | `zzz_temp_title`, `zzz_temp_artists`, `zzz_temp_owner`, `zzz_temp_note` |
| Filter fields | `zzz_filter_` prefix | `zzz_filter_Status`, `zzz_filter_Label`, `zzz_filter_priority` |
| Sort fields | `zzz_` + `_Sort` suffix | `zzz_Owner_Sort`, `zzz_Title_Sort`, `zzz_ISRC_Sort` |
| Sort evaluation | `zz_SORT_` prefix | `zz_SORT_Priority_Evaluation`, `zz_SORT_Priority_Index` |
| Pivot/relationship | `zv_` prefix | `zv_Pivot_gt`, `zv_Utility_gt` |
| Layout-scoped global text | `zl_` prefix | `zl_WebViewerURL_gt`, `zl_Utility1_g` |
| Constant fields | `always_` prefix | `always_1`, `always_0`, `always_active`, `always_artist`, `always_writer` |
| Global fields | `G_` or `g` prefix | `G_ClipBoard`, `G_ThisDate`, `gAccountName`, `gExportFolder` |
| Container (global ref) | `zzz_` prefix + `_gr` suffix | `zzz_CompanyLogo_gr`, `zzz_SecondLogo_gr` |
| Stored calc suffix | `_stored` | `Track_Status_stored`, `Song_Cleared_stored` |
| Calculated text suffix | `_calc` | `Track_display_calc`, `Artists_list_calc` |
| Flag fields | `_flag` suffix or descriptive | `Instrumental_flag`, `Move_to_excluded_flag`, `Song_Cleared` (0/1) |

**The `zzz_` system is central to Hero's architecture.** It pushes utility/system fields to the bottom of field lists in FM's alphabetical display. The `zz_` prefix (two z's) is used for sort-specific evaluation fields. The `z` prefix (single z) is for audit fields.

### Layouts

Greg's actual pattern:
- Descriptive names, NO table prefix convention: `Priority List`, `Clearance List Tracks`, `Clearance List Shares`, `Interface`, `Writers List`, `Add New Track Form`
- Card/popup layouts: descriptive with context: `Tracks Clearance Fields NEW`, `SHares Preview Card`
- Dev layouts: `DEV Interface`

### Scripts

Greg's actual naming:
- **Sentence case / natural language** — NOT strict Title Case
- Module prefix with dash: `Clearance - Open Track Card`, `Priority - On Layout Enter`, `Writers - on layout enter`
- Trigger indicator: `[TRIGGER]` suffix or `on` prefix: `Halt TAB Order [TRIGGER]`, `on Clearance layout enter`
- PSOS prefix: `PSOS - Get Dash Figures`, `PSOS - log User Screen Prefs (params)`
- Parameters in name: `Writers - Create Email - param(email)`, `Select Mode - Tracks [param]`
- Server scripts in dedicated folder: `Server Scripts/`
- Button scripts in dedicated folder: `Scripts for Buttons/`

### Custom Functions

- TitleCamelCase: `Context`, `DeclareVariables`, `ErrorDescription`

## Calculation Code

### Let() Formatting

Greg uses tabs for indentation and underscore-prefixed short variable names:

```
Let([
			_p = "New";
			_p2 = $priority_number
			];

			ExecuteSQL ("
			SELECT COUNT (" & GetSQLFieldName ( Tracks::Track_UID ) & ")
			FROM " & GetSQLTableName ( Tracks::zzz_record_number ) & "
			WHERE " & GetSQLFieldName ( Tracks::priority_number ) & " = ?
			AND " & GetSQLFieldName ( Tracks::Track_Status_stored ) & " = ?
			 "; "" ; ""; _p2; _p )

			)
```

- **Hard tabs** for all indentation inside calculations
- Let variable names: `_p`, `_p2`, `_l` — short, underscore-prefixed
- Multiple levels of tab indentation used liberally

### Operators

- Uses `≠` and `≥` (Unicode operators): `$lastError ≠ 0`, `$counter ≥ $count`
- Spaces around operators: `$counter + 1`, `$loop_counter +1` (slightly inconsistent spacing on increment)
- String comparison with `=`: `$param = "Tracks"`, `$Type = "Form"`

### String Concatenation

- Space on both sides of `&`: `"logon " & Get(HostName) & " " & Get(AccountName)`
- Multi-line SQL strings use `&` continuation

### Boolean Values

- Uses `1` and `0` directly (not `True`/`False`): `Set Variable [ $$File_is_local ; 1 ]`
- Empty string `""` used to clear/reset: `Set Field [ Publishers::Excluded_flag ; "" ]`

### Function Spacing

- Spaces inside function parens: `Get ( AccountName )`, `PatternCount ( Get (ApplicationVersion); "server" )`
- Slightly inconsistent — sometimes `Get(LastError)`, sometimes `Get ( LastError )`
- Both are acceptable; don't force one style

## Script Patterns

### Standard Script Opening

Most scripts begin with this boilerplate:

```
Allow User Abort [ OFF ]
Set Error Capture [ ON ]
```

Sometimes reversed order (`Set Error Capture` first). Either order is acceptable.

### Script Documentation Header

Greg's older scripts use a comment-based header:

```
# Created:      Greg Zielinski - 24-07-2021
# History:

# Purpose: [one-line description]

# Parameters: [parameter description]
```

**Note:** Not all scripts have this header. Many newer scripts jump straight into `Allow User Abort` / `Set Error Capture`. When generating new scripts, include the header for complex scripts but skip it for simple button handlers.

### Parameter Handling

Two patterns observed:

**Simple parameter:**
```
Set Variable [ $param ; Get(ScriptParameter) ]
If [ IsEmpty($param) ]
    Exit Script [ 1 ]
End If
```

**DeclareVariables (multi-param):**
```
Set Variable [ $var ; DeclareVariables ( Get(ScriptParameter) ) ]
```

### Loop Pattern

Greg's standard loop uses a counter (not `Go to Record/Next`):

```
Set Variable [ $loop_counter ; 1 ]
Set Variable [ $loop_total ; Get(FoundCount) ]
Loop
    Go to Record/Request/Page [ $loop_counter ]
    Exit Loop If [ $loop_counter ≥ $loop_total ]
    Set Variable [ $loop_counter ; $loop_counter + 1 ]
End Loop
```

### Error Handling Pattern

```
[risky step]
If [ Get(LastError) ≠ 0 ]
    Show Custom Dialog [ Title: "Error" ; Message: [description] ]
    Exit Script [ 1 ]
End If
```

Also uses `Beep` before error dialogs for user attention.

### PSOS Pattern

Server-delegated scripts check application version and delegate:

```
If [ not PatternCount(Get(ApplicationVersion); "server") ]
    Perform Script on Server [ ... ]
    Exit Script [ 1 ]
End If
[server-side logic]
```

### Commit Before Navigate

`Commit Records/Requests [ With dialog: OFF ]` commonly appears before layout changes or at script start after `Set Error Capture`.

### Card/Popup Windows

Uses `Go to Related Record` with `New window` option for card views. Uses `Close Window [ Name: "..." ]` for cleanup.

### Section Headers

Uses doubled comment lines as visual section dividers:

```
#  TRACKS
#  TRACKS
```

or

```
#  OWNERS
#  OWNERS
```

### Freeze Window

`Freeze Window` used before multi-step layout navigation to prevent screen flicker.

### Exit Script

- `Exit Script [ 1 ]` used as a general-purpose exit (not just error)
- `Exit Script [ "ok" ]` for success indication to callers
- `Exit Script []` (empty) for clean exits without result

## Control Structures

### If Statements

- No `// Else` comment convention used (departs from default template)
- Direct `Else` without annotation

### Validation Pattern

Consistent pattern for user-facing validation:

```
If [ IsEmpty($required_value) ]
    Show Custom Dialog [ Title: "Validation" ; Message: "Please select [thing] first" ]
    Exit Script [ 1 ]
End If
```

### Confirmation Dialog Pattern

```
Show Custom Dialog [ Title: "Question" ; Message: "..." ; Default Button: "Cancel" ; Button 2: "Yes" ]
If [ Get(LastMessageChoice) = 1 ]
    Exit Script [ 1 ]
End If
```

Button 1 = Cancel (exit), Button 2 = Confirm (proceed).

## Commenting

- Inline comments use `//`
- Use `MARK: SectionName` (disabled `# (comment)`) at the top of logical sections within a long script (similar to code folding markers) — no `//` prefix
- Comment the _why_, not the _what_ — avoid restating what the step obviously does
- In Case/If branches, use `// Else` as a visual separator before Else steps
- Script-level comments describe purpose, expected context, parameters, and return values

### HR format comment syntax

FileMaker's `# (comment)` step has two states — enabled and disabled — and both appear in HR format. The syntax convention is:

| HR syntax | Step state | XML | Use for |
|-----------|-----------|-----|---------|
| `# text` | Enabled (`enable="True"`) | `<Step enable="True" id="89" name="# (comment)"><Text>text</Text></Step>` | Visible annotations, PURPOSE line |
| `#// text` | Disabled (`enable="False"`) | `<Step enable="False" id="89" name="# (comment)"><Text>MARK: SectionName</Text></Step>` | Section markers, structural dividers — use `MARK: SectionName` without any `//` prefix |
| `#` | Enabled, blank | `<Step enable="True" id="89" name="# (comment)"/>` | Visual blank line between sections |

The `//` prefix on a disabled comment is a visual convention borrowed from other languages — it signals "this step is off" in the same way `//` means a commented-out line. `// MARK:` headers always use the disabled form `#//` so they appear greyed out in the Script Workspace and do not clutter the active comment layer.

This distinction matters when HR code is pasted into FileMaker or converted by the webviewer's HR→XML converter: `# text` and `#// text` must produce different `enable` attribute values.

### Script documentation structure

Every script follows this documentation pattern at the top:

**1. PURPOSE line** — always the very first step, a single `# (comment)` with `PURPOSE: ...`:
```xml
<Step enable="True" id="89" name="# (comment)">
  <Text>PURPOSE: One-line description of what this script does.</Text>
</Step>
```

**2. Doc block** (when additional detail is needed) — a disabled `Insert Text` step targeting `$README`. The variable is never set (step is disabled) but the name makes it self-documenting. Use `&#xD;` for line breaks:
```xml
<Step enable="False" id="61" name="Insert Text">
  <SelectAll state="False"/>
  <Text>PARAMETERS: $param — JSON with keys ...&#xD;&#xD;RETURNS: ...</Text>
  <Field>$README</Field>
</Step>
```

**3. Blank lines** — empty self-closing `# (comment)` steps used as visual separators between logical sections:
```xml
<Step enable="True" id="89" name="# (comment)"/>
```

### Example pattern

FileMaker is a copy-paste environment. Scripts that are meant to be called from other scripts should include a usage example so the caller can copy and adapt it.

**Placement depends on content:**

- **Bottom of script** (after `Exit Script`) — use a disabled step showing the call pattern. `Exit Script` prevents execution even if examples are somehow enabled:
  ```xml
  <Step enable="True" id="103" name="Exit Script"/>
  <Step enable="True" id="89" name="# (comment)">
    <Text>Example of how to call this script</Text>
  </Step>
  <Step enable="False" id="1" name="Perform Script">
    <Calculation><![CDATA[JSONSetElement ( "{}" ; ... )]]></Calculation>
    <Script id="0" name="Script Name"/>
  </Step>
  ```

- **Top of script** (inline with documentation) — use a disabled `Set Variable` with an `$example_[description]` name. Being at the top makes it immediately visible; being disabled means it never executes:
  ```xml
  <Step enable="False" id="141" name="Set Variable">
    <Name>$example_basicCall</Name>
    <Value><Calculation><![CDATA[JSONSetElement ( "{}" ; [ "key" ; $value ; JSONString ] )]]></Calculation></Value>
  </Step>
  ```

## Scripting & Organization

### Script Naming

- Title Case with spaces: `Process Invoice`, `Validate Line Items`
- Include parameters in the name when it aids discoverability: `Navigate To Layout ( layoutName )`
- Prefix with a functional area when scripts are shared: `Email: Send Confirmation`

### Folder Structure (baseline)

```
Application/
  Startup
  Shutdown
  Menus
  Shared/          ← solution-wide utility scripts
User Interface/
  <Layout Name>/   ← one folder per layout, named to match
```

- Scripts specific to a single layout go in that layout's folder
- Reusable scripts shared across layouts/contexts go in `Shared/`
- Extend the structure as needed — this is a minimum baseline

### Script Step Conventions

- Use `Exit Script [ Result: ... ]` to return values, not global variables, wherever possible
- Always `Commit Records/Requests` before navigating away from a record context if edits may have occurred
- Error capture (`Set Error Capture [ On ]`) should be paired with explicit error checking immediately after the risky step
- Comment the expected script parameter format at the top of any script that accepts parameters

## Empty Strings / Null Values

- `""` for empty string — used to clear fields: `Set Field [ field ; "" ]`
- `IsEmpty()` for testing: `IsEmpty($param)`, `IsEmpty(Tracks::Added_to_tracker)`
- `1` and `0` for boolean flags, not True/False

## ExecuteSQL Usage

Greg uses `ExecuteSQL` with `GetSQLFieldName()` and `GetSQLTableName()` for rename-safe SQL:

```
ExecuteSQL ("
    SELECT COUNT (" & GetSQLFieldName ( Tracks::Track_UID ) & ")
    FROM " & GetSQLTableName ( Tracks::zzz_record_number ) & "
    WHERE " & GetSQLFieldName ( Tracks::priority_number ) & " = ?
"; "" ; ""; $paramValue )
```

- Uses `?` placeholders for parameters
- SQL keywords in UPPERCASE: `SELECT`, `COUNT`, `FROM`, `WHERE`, `AND`
- Field/table references via `GetSQLFieldName()` / `GetSQLTableName()`

---

## Reserved Words

Avoid using FileMaker reserved words as field, variable, table, or script names. Common ones to avoid:

`Status`, `Error`, `Name`, `Date`, `Time`, `Count`, `List`, `Max`, `Min`, `Sum`, `Average`

Disambiguate with context: `Track_Status`, `Live_Date`, `Owner_name`

## Summary of Deviations from Default Template

| Convention | Default (filemakerstandards.org) | Greg's Actual Style |
| --- | --- | --- |
| Let variables | `~` prefix | `_` prefix |
| Global variables | `$$ALL.CAPS.DOTS` | `$$snake_case` or `$$camelCase` |
| Tables | TitleCase single word | Underscore_Separated |
| Fields | camelCase | Mixed — snake_case dominant, some camelCase |
| Layouts | `Table: Name (type)` | Descriptive names, no table prefix |
| Scripts | Strict Title Case | Sentence case, module prefix with dash |
| Booleans | `True` / `False` | `1` / `0` |
| Else comments | `// Else` before each branch | Not used |
| Script doc | `PURPOSE:` as first line | `# Created:` / `# Purpose:` header block |
